## Os dois tipos de webhook

São cotas **separadas**, para coisas diferentes. Se você está conectando uma plataforma de checkout ao CRM, o que conta é a cota de **entrada**.

| | Webhooks de entrada (inbound) | Webhooks outbound |
|---|---|---|
| Direção | Sistema externo → Vistum | Vistum → sistema externo |
| Para que serve | Receber vendas, leads e eventos de Hotmart, Kiwify, Eduzz, Braip, Monetizze, Perfect Pay, Kirvano, n8n, formulários e qualquer sistema que faça POST HTTP | Notificar o seu sistema quando algo acontece no CRM (lead criado, mensagem recebida, card movido) |
| Onde se configura | Automações → gatilho **Webhook recebido** (gera URL + token) | Configurações → Desenvolvedor → **Webhooks** |
| Essencial | **5** | — |
| Crescimento | 20 | 5 |
| Sob consulta | Ilimitado | Ilimitado |
| Documentação | [Webhooks de Entrada](/webhooks/inbound) | [Webhooks (outbound)](/webhooks/overview) |

<Check>
  O plano **Essencial** integra plataformas de checkout. Cada automação com gatilho **Webhook recebido** consome 1 da cota de webhooks de entrada — com 5, dá para conectar Hotmart, Kiwify, Eduzz e ainda sobra folga para fluxos de reembolso ou um formulário externo.
</Check>

## Como os limites funcionam

Os limites são aplicados em **soft enforce**: o workspace continua funcionando normalmente, mas novas criações são bloqueadas quando o limite é atingido.

Por exemplo, ao atingir o limite de contatos, a API retorna `402 Payment Required` com `code: "CONTACT_LIMIT_REACHED"`, mas os contatos existentes continuam acessíveis:

```json
{
  "error": "Contact limit reached for this workspace's plan",
  "code": "CONTACT_LIMIT_REACHED"
}
```

O limite de negócios devolve `DEAL_LIMIT_REACHED` no mesmo formato. Programe suas checagens contra o campo `code` — o texto de `error` pode variar entre endpoints. Veja todos os códigos em [Erros](/errors).

<Note>
  O limite de **etapas por pipeline** é por pipeline individual, não a soma de todos os pipelines do workspace.
</Note>
